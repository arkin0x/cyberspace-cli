### Mock Backend Limitations

Current implementation uses `_generate_mock_stark_proof()`:
- Generates deterministic hash-based proof objects
- Does NOT provide cryptographic soundness
- Does NOT detect tampering (test skipped for this reason)
- Proof size: ~77 bytes (vs ~30 KB expected for real STARK)

**This is intentional** - the interface is designed to be backend-agnostic. Production deployment would replace with:
- `cairo-lang` (StarkWare)
- `winterfell` (Meta/Facebook)
- `plonky3` (Polygon)
- Custom Rust implementation

## Key Decisions Made

### 1. Backend Agnosticism ✅ CONFIRMED

The existing implementation follows the design spec's recommendation:
```python
# From zk_cantor.py docstring:
# "The interface is designed to be backend-agnostic - swap the implementation,
# keep the API."
```

**Decision:** Continue with this pattern. Build real STARK backend as pluggable module.

### 2. Optional `zk` Tag ✅ CONFIRMED

Design spec recommended optional `zk` tag (not new action type):
```json
{
  "kind": 3333,
  "tags": [
    ["A", "hop"],
    ["proof", "<cantor_root_hex>"],  // Always required
    ["zk", "<stark_proof_hex>"]       // Optional optimization
  ]
}
```

**Decision:** Follow this pattern for backward compatibility.

### 3. Work Equivalence Preservation ✅ CONFIRMED

Current implementation maintains work equivalence:
- Prover still computes full Cantor tree (`build_hyperspace_proof(leaves)`)
- ZK proof is **in addition to** standard proof, not replacement
- Verifier can fall back to standard verification if no `zk` tag

**Decision:** This is correct. ZK is an efficiency layer, not a replacement.

## Blockers Encountered

### 1. Tampering Detection (Known Limitation)

**Problem:** Mock backend doesn't detect proof tampering.

**Test:** `test_verify_detects_wrong_result` is skipped.

**Resolution:** Requires real STARK backend. Defer to Phase 2.

### 2. Proof Size Discrepancy

**Current:** ~77 bytes (mock)
**Target:** ~30 KB (real STARK for h33)

**Impact:** Can't validate Nostr event fit until real backend.

**Resolution:** Accept for PoC. Benchmark with real backend in Phase 2.

**Design spec calls for:**

**Current:** None implemented.

**Resolution:** Implement in Session 2-3.

## Performance Measurements

### Current (Mock Backend)

| Operation | Time | Notes |
|-----------|------|-------|
| `prove_single_cantor_pair(42, 17)` | <1 ms | Overhead dominated |
| `prove_cantor_tree([12345, 100, 101, 102])` | <1 ms | Mock proof generation |
| `verify_single_cantor_pair(z, proof)` | <1 ms | Trivial structure check |

### Expected (Real STARK Backend)

Based on winterfell/plonky3 benchmarks (Design §5):

| Tree Height | Proof Gen Time | Verification Time | Proof Size |
|-------------|----------------|-------------------|------------|
| h10 | ~100 ms | ~1 ms | ~15 KB |
| h20 | ~1-5 sec | ~2 ms | ~20 KB |
| h30 | ~10-30 sec | ~3 ms | ~25 KB |
| h33 | ~1-3 min | ~4 ms | ~30 KB |

**Gap Analysis:**
- Proof generation: 10-100× slower than plain Cantor (within target of <10× for h33)
- Verification: ~4 ms vs target <10 ms ✅
- Size: ~30 KB vs target <100 KB ✅
