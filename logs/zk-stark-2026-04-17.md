### Key Decisions Made

1. **STARK over SNARK:** ZK-STARK selected for post-quantum security and no trusted setup (critical for long-term protocol integrity)

2. **Library choice:** winterfell for Phase 1 (Python support, easier prototyping), plonky3 for Phase 2 (production performance)

3. **Circuit design challenge:** Direct Cantor arithmetization requires ~43 billion constraints for height-33 tree — **infeasible** with current STARK technology

4. **Proposed solution:** Hybrid approach using STARK-friendly hash (Poseidon) for the arithmetized circuit, while preserving Cantor root in the protocol for backward compatibility

5. **Integration pattern:** Optional `zk` tag in existing hop events (not new action type) for backward compatibility

6. **Performance estimates:**
   - Proof size: ~30 KB (~60 KB encoded) — ✅ fits in Nostr event
   - Verification time: ~4 ms — ✅ under 10 ms target
   - Proof generation: ⚠️ challenging to meet <10× overhead target

### Blockers Encountered

1. **Circuit size problem:** The most significant blocker is that a direct Cantor tree STARK would require ~43 billion constraints for height-33, which is 10,000× beyond current STARK capabilities.

   **Mitigation paths:**
   - Use Poseidon hash for the STARK circuit (maintains binding, loses exact Cantor formula)
   - Recursive proof composition (prove subtrees, compose)
   - Plonky3's optimized prover (may handle larger circuits than winterfell)

2. **Discovery: PoC already exists!** Found existing implementation:
   - `src/cyberspace_core/zk_cantor.py` — Python mock/STARK interface
   - `src/cyberspace_core/zk_cantor/` — Rust + PyO3 module using winterfell
   - `tests/test_zk_cantor_tree_future.py` — Comprehensive test suite (22 tests)
   - Currently at **Phase 2 (Single-Axis Tree)** stage per hourly task cycle

3. **Test failures in tree-level proofs:** The Rust implementation (via `zk_cantor/__init__.py`) uses a different interface than the Python mock. Tests expect `ZKCantorProof` class but get dict. This is an **integration issue**, not a fundamental blocker.

## Notes

**Critical insight from RATIONALE.md §7:** Work equivalence is the "key property" of Cyberspace. Any ZK solution **must preserve** the requirement that provers do full Cantor tree computation. ZK only optimizes verification, not generation.

**Quote from canton.astro (line 38-41):**
> "The proof requires real computational work to create. Verification costs the same amount of work in the current implementation, though future zero-knowledge proof systems could make verification nearly free while maintaining the same security guarantees."

This confirms ZK-STARKs are an **expected evolution** of the protocol, not a deviation.
